# v3.1 — Foundation: data model + auth re-gate

**Goal.** Land the schema and authorization changes that every later phase depends on. No new user-facing features yet — this is plumbing.

**BRD references.** IV.2.2, IV.2.3, IV.4.1, IV.5.1.1 (projectType), IV.6 (status), I (TCNL/KSV definitions), and the audit's blocking items B1, B2, B7.

## Decisions locked (2026-05-04)

- **Status enum migration:** **wipe demo data + re-seed** with new values. Production-data migration deferred (no production data exists).
- **TCNL / KSV title assignment in seed:** the seed script will set `functionalTitle = 'KSV'` on `dev.duy` and `'TCNL'` on `dev.khang` so v3.2's close-workflow has live test users.
- **Auth provider:** Supabase Auth stays (B4 out of scope). `User.functionalTitle` is added regardless because v3.2 needs to identify TCNL/KSV regardless of how login happens.

## Tasks

### Schema changes

`backend/prisma/schema.prisma`

- [ ] **Replace `ProjectStatus` enum.**
  ```prisma
  enum ProjectStatus { ACTIVE PAUSED CLOSED }
  ```
  Backfill SQL (one-shot in a Prisma migration):
  ```sql
  update projects set status = 'ACTIVE'  where status in ('INITIATION','PLANNING','IN_PROGRESS','AT_RISK');
  update projects set status = 'CLOSED'  where status = 'DONE';
  ```

## Migration steps (run order)

```bash

cd backend
npx prisma migrate dev --create-only --name v3.1_foundation
# 2. hand-edit the generated SQL to add the value-mapping UPDATEs (status enum)
# 3. apply
npx prisma migrate deploy
# 4. backfill personnel-to-rows
npx tsx src/db/migrations-v3.1/02-personnel-to-rows.ts
# 5. re-seed (optional — wipes data)
npm run seed
# 6. verify
npx tsx src/db/smoke-test.ts
npx tsx src/db/e2e-test.ts
```
