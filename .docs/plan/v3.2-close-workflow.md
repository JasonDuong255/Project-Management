# v3.2 — Close workflow + auto health + lock-when-closed

**Goal.** Implement the BRD's IV.6 multi-step project close, the IV.5.6 deadline alerts, and IV.6's "no edits when closed" rule. Together this closes audit blockers B3, B5, B6 and the back half of B2.

**Depends on.** v3.1 must be merged first (needs the new status enum, `functionalTitle`, `project_close_requests` table).

## Tasks

### Workflow endpoints

`backend/src/modules/close-workflow/` (new module)

- [ ] **`POST /api/projects/:id/pause`** — QLDA only (PM or coordinator). `status: ACTIVE → PAUSED`, write `pausedAt`, log `PROJECT_PAUSED`.
- [ ] **`POST /api/projects/:id/resume`** — QLDA only. `PAUSED → ACTIVE`, log `PROJECT_REOPENED_FROM_PAUSE`.
- [ ] **`POST /api/projects/:id/close-requests`** — QLDA submits a close request. Body: `{ note?: string }`. Creates a row in `project_close_requests` with `ksvDecision='PENDING'`. Sends notification to all KSVs. Logs `CLOSE_REQUESTED`.
- [ ] **`PATCH /api/projects/:id/close-requests/:requestId/ksv`** — KSV decision. Body: `{ decision: 'APPROVED' | 'REJECTED', reason?: string }`. Sets `ksvDecision`, `ksvDecidedById`, `ksvDecidedAt`, `ksvRejectReason`. If APPROVED: notify all TCNLs. If REJECTED: notify the requester. Logs `CLOSE_APPROVED_KSV` or `CLOSE_REJECTED_KSV`.
- [ ] **`PATCH /api/projects/:id/close-requests/:requestId/tcnl`** — TCNL decision. Same pattern. APPROVED → set `project.status = 'CLOSED'`, write `closedAt`, log `CLOSE_CONFIRMED_TCNL`. REJECTED → notify, log `CLOSE_REJECTED_TCNL`.
- [ ] All endpoints validate role + project status (must be `ACTIVE` or `PAUSED` to start a close request; `PENDING` request must exist before KSV/TCNL act).

### Lock-when-closed middleware (B6)

- [ ] New middleware `requireProjectNotClosed` in `backend/src/middlewares/project-lock.ts`. Reads `req.params.projectId`, looks up status, throws 403 if `CLOSED` (allow-list: GET endpoints + close-request endpoints themselves).
- [ ] Apply to every project-mutation route in `routes.ts`.

### Auto health + alerts (B2 part 2 + B5)

`backend/src/jobs/` (new folder; we don't have a job runner yet)

- [ ] Add a lightweight scheduler. Options:
  - **A. node-cron in the same Express process.** Simple. Runs every 6 hours. Acceptable for one-instance deployment.
  - **B. Supabase pg_cron + DB-only logic.** Schedules in Postgres; no Node code. Cleaner for HA but harder to debug.

  Recommend A for v3.2; revisit if scaled out.
- [ ] **Job: `recompute-project-health`** (every 6 h, or on plan-item save):
  ```
  for each ACTIVE project:
    plan_items_overdue   = count where endDate < today  AND progress < 100
    plan_items_near_due  = count where endDate ∈ [today, today+4d] AND progress < 100
    if plan_items_overdue > 0: project.health = AT_RISK
    elif plan_items_near_due > 0: project.health = NEEDS_REVIEW
    else: project.health = STABLE
  ```
- [ ] **Job: `notify-deadline-alerts`** (daily):
  - For each task that crosses the 4-day-before-deadline threshold today, insert a `Notification` row for every TTK member of the project, kind `TASK_DEADLINE_NEAR`.
  - For each task that just went past its end date with progress < 100, insert `TASK_OVERDUE` notifications.
- [ ] Make the scheduler optional (env var `RUN_SCHEDULER=true`) so dev environments can opt out.

### Notifications (foundation)

- [ ] New table `notifications` (schema in `audit/data-model-delta.md`).
- [ ] Endpoints:
  - `GET /api/notifications` — current user's, with unread filter.
  - `POST /api/notifications/:id/read` — mark read.
- [ ] Email integration: when a notification is `CLOSE_REQUEST_RECEIVED` / `CLOSE_REJECTED` / `CLOSE_APPROVED`, also send an email. Use `nodemailer` with a **pluggable transport**: dev mode writes to `console.log` (default, no infra dependency); prod mode reads SMTP config from `backend/.env` (`SMTP_HOST`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`). Switch via `EMAIL_TRANSPORT=console|smtp` env var. (BRD III.1.2.5: "Hệ thống thông báo trên ứng dụng và gửi mail".)
