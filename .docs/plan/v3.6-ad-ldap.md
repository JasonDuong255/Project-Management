**Goal.** Replace (or federate) Supabase email-password auth with AITS AD/LDAP per BRD IV.1 + VIII.3.

**Depends on.** Independent of feature phases. Requires **infra access** to the AITS AD/LDAP server — coordinate with the IT team early.

## Approach options

### Option A: Federate via Supabase Auth + SAML

Supabase Auth supports SAML 2.0 SSO. Configure AITS AD as the SAML IdP (most ADs expose SAML via ADFS).

- **Pros:** keep all existing Supabase Auth code; users see AD login; sessions still managed by Supabase; FE doesn't change much.
- **Cons:** SAML setup with ADFS is non-trivial; needs an AD admin's help.

### Option B: Replace Supabase Auth with passport-ldapauth

Drop Supabase Auth client-side, do username/password against LDAP directly via the backend. Issue our own JWTs.

- **Pros:** simpler infra; talks LDAP straight; lower vendor lock-in.
- **Cons:** the backend now owns sessions + JWT refresh; more code to write; lose Supabase's password reset / MFA features.

### Option C: Hybrid — AD-managed users mirrored into Supabase

A nightly job pulls AD users into Supabase via admin API. Login is still email-password but the password is provisioned from AD.

- **Pros:** minimal code change.
- **Cons:** password sync is brittle and security-risky (AD password leaving AD); not really AD auth.

**Recommendation: Option A** if AITS has ADFS available, **Option B** otherwise.
