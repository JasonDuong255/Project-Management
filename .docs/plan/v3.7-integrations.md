# v3.7 — Integrations: HRM, KTQT, a.Office

**Goal.** Wire the three external systems the BRD requires (II.2.4, VIII.2). Build mock layers first so the FE can be developed against fake data, then swap to real endpoints.

**Depends on.** Mostly independent. v3.1 if the integrations need new fields.

## Decision locked (2026-05-04)

**Build mocks early, swap to real later.** Adapter stubs land alongside v3.3's personnel work so the FE's AITS-personnel picker, KTQT validation, and a.Office worklog flows have realistic shapes from day one. The real client implementations swap in via `INTEGRATION_MODE=real` once endpoints are reachable.

## Architecture

`backend/src/integrations/<name>/`

```
backend/src/integrations/
  hrm/
    hrm.adapter.ts         # interface
    hrm.mock.ts            # local fake — used in dev
    hrm.client.ts          # real HTTP client — used in prod (env-flag)
    hrm.sync.ts            # scheduled job
  ktqt/
    ktqt.adapter.ts
    ktqt.mock.ts
    ktqt.client.ts
  aoffice/
    aoffice.adapter.ts
    aoffice.mock.ts
    aoffice.client.ts
    aoffice.push.ts        # outbound: worklogs to a.Office
    aoffice.pull.ts        # inbound: personal plans + actuals
```

`env` switch: `INTEGRATION_MODE=mock|real`. Default `mock` until prod credentials available.

## HRM integration

- [ ] **Adapter:** `getEmployees() → HrmEmployee[]`, `getEmployeeById(id)`, `searchByName(q)`.
- [ ] **Sync job** (daily): pull all active employees, upsert into `profiles`. Lock fields (employeeCode, unit, title) so they cannot be edited locally — they always sync from HRM.
- [ ] **FE:** when picking AITS personnel, the "Add personnel" modal calls `GET /api/users?source=hrm` which proxies to the HRM adapter (or to local profiles synced from HRM — same result).
- [ ] BRD IV.5.1.2.3: AITS fields locked when sourced from HRM. Implement via field-level read-only flag.

## KTQT integration

- [ ] **Adapter:** `getProjectHourCap(projectCode) → number | null` (null when no business plan exists).
- [ ] **Validation hook**: in `POST /api/projects/:id/allocations`, before upsert, sum project's allocations + the new value; reject with 422 if `> capFromKtqt && projectType in (CONTRACT, FEASIBILITY)`. (BRD III.1.2.2.3.2.)
- [ ] **FE:** show the cap and current usage on the WorkloadPage tab; surface the rejection reason inline.

## a.Office integration

This one is the heaviest — bidirectional sync.

- [ ] **Outbound (push worklogs):**
  - On every `addWorklog`, queue an a.Office push job (or push synchronously and store result).
  - Field mapping: { date, projectCode, taskName, hours, memberCode, content } → a.Office worklog.
  - On success set `worklogs.syncedToAOfficeAt`. Retry on transient failures.
- [ ] **Inbound (pull personal plans + actuals):**
  - Daily job: for every active user, fetch a.Office plan + actuals.
  - Store in new tables: `aoffice_personal_allocations` (plan), `aoffice_personal_actuals` (executed). Keyed by (userId, month) and category (DBHĐ, RQ, CRQ, Khác, PJ).
  - The Resources view (BRD IV.5.3.3) reads from these tables.
- [ ] **Reconciliation:** if a worklog was edited locally, mark it for re-push. Maintain a sync queue with last-attempted + status.
