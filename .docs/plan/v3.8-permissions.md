# v3.8 — Permission groups (sơ-đồ-cây UI) — DEFERRED

**Goal.** Replace the 4-value role enum with a permission-group system per BRD IV.2 ("Quản lý phân quyền: dạng sơ đồ cây") + VI ("Admin chủ động phân quyền").

**Status: deferred.** The 4-role enum (`PMO`/`ADMIN_HC`/`PM`/`DELIVERY_MEMBER`) covers BRD's named roles cleanly today via `User.functionalTitle` (TCNL/KSV) introduced in v3.1. Add this phase only when:

- A real customer requires fine-grained per-function authorization that the 4 roles + functionalTitle can't express, or
- A new role appears that maps cleanly to neither the 4 enum values nor the `functionalTitle` set.

## When you do build it

### Schema

```prisma
model PermissionGroup {
  id           String  @id @default(cuid())
  name         String  @unique
  description  String  @default("")
  isSystem     Boolean @default(false)
  functions    PermissionGroupFunction[]
  users        UserPermissionGroup[]
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}
model PermissionGroupFunction {
  groupId      String
  functionKey  String
  group        PermissionGroup @relation(fields: [groupId], references: [id], onDelete: Cascade)
  @@id([groupId, functionKey])
}
model UserPermissionGroup {
  userId       String  @db.Uuid
  groupId      String
  user         User             @relation(fields: [userId], references: [id], onDelete: Cascade)
  group        PermissionGroup  @relation(fields: [groupId], references: [id], onDelete: Cascade)
  @@id([userId, groupId])
}
```

### Function keys

The "tree" displayed in the UI comes from a constant tree of function keys, e.g.:

```
projects:create
projects:edit-info
projects:close
plan-items:create
plan-items:edit
plan-items:delete
worklogs:create-self
worklogs:create-others
risks:create
risks:edit
risks:delete
documents:upload
documents:edit
documents:delete
catalogs:edit-system
catalogs:edit-non-system
admin:users:view
admin:users:edit
admin:permissions:edit
reports:export-weekly
reports:export-detailed-plan
```

Define this tree in one constant file shared between FE and BE.

### Authorization rewrite

Replace `requireRoles([...])` with `requirePermission('projects:create')`. The middleware looks up the user's groups, unions their function keys, and checks membership.

The 4-role enum becomes a "bootstrapping default" for new users — they get auto-added to `permission_groups` matching their role.

### FE

- New page or panel under `/admin/permissions`:
  - Left: list of permission groups (with create/edit/delete).
  - Right when group selected: function tree with checkboxes; save updates `permission_group_functions`.
  - Bottom: list of users in the group with add/remove.
- Tree component options: `react-arborist` or roll a small one. The tree is small (<50 nodes), so nothing fancy needed.

### Migration

- All existing users get auto-assigned to a "default" group matching their current `role`.
- `requireRoles` middleware stays as a fallback for unmigrated routes; gradually replace.
